# Reglas para Agentes de IA

Si eres un agente de IA interactuando con este repositorio, debes acatar las siguientes reglas:

1. **Metodología KDD Obligatoria**: Este repositorio utiliza Knowledge-Driven Development. Antes de escribir código, debes leer los contratos en `knowledge/contracts/` y la spec normativa de los nodos OKF en `knowledge/OKF-SPEC.md`.
2. **Carga la Skill Local**: Tienes disponible la skill `kdd-okf-ccdd-hybrid` en el directorio `.agents/skills/`. Debes adoptarla en tu contexto para entender cómo generar y validar los contratos híbridos OKF-CCDD.
3. **No dupliques contexto**: Utiliza enlaces de markdown relativos a `knowledge/` cuando necesites explicar el porqué de una implementación.
4. **Validación Determinista (dos niveles)**: la referencia canónica completa (niveles, gate multi-lenguaje, export, `tc-tests-frozen`) está en [knowledge/validacion.md](../knowledge/validacion.md) — léela antes de validar; esta regla no la duplica. Resumen: nivel 1 (obligatorio) = `python scripts/validate_contracts.py knowledge/contracts` (incluye el sello obligatorio `tests_sha256` del oráculo; sellar con `--hash`) + `python scripts/validate_specs.py specs` + `python scripts/validate_okf.py knowledge` (estructura/frontmatter de nodos OKF) + `python scripts/lint_ascii.py scripts` + `python scripts/validate_rules.py <dir>` (rule contracts, capa opcional) + `python scripts/validate_skills.py skills .agents/skills` (skills de agente, capa opcional) + `python scripts/validate_changelog.py` (coherencia CHANGELOG↔reportes, capa opcional) + `python scripts/validate_ux_page.py examples/ux-page` (UX/accesibilidad mecánica sobre paginas HTML autocontenidas, capa opcional) + `python scripts/validate_diagrams.py <dir>` (diagramas Mermaid flowchart verificados contra un .diagram-contract.json declarativo, capa opcional) + `python scripts/validate_test_commands.py <contracts_dir> <repo_root>` (corre el test_command real de cada contrato y falla si algun exit code no es 0, Nivel 1 obligatorio) + `python scripts/scan_secrets.py <dir>` (escaneo determinista de credenciales filtradas por prefijo de proveedor conocido, capa opcional) + el `test_command` del contrato, local y en CI (matriz dual-OS, suite 2x); nivel 2 (opcional) = gate CCDD vía MCP `ccdd-complexity` (`lint_task_contract`, `run_integration_gate`) sobre el export de `scripts/export_gate_contract.py`. Ningún contrato se considera terminado hasta que pase el nivel 1.
5. **Precedencia del Budget**: ver [knowledge/validacion.md](../knowledge/validacion.md) — con gate manda su config firmada (el `budget` del frontmatter solo puede ser <=); sin gate, el `budget` es declarativo y el validador solo verifica su presencia.
6. **Ciclo de Vida del Contrato**: `draft` -> `validated` -> `implemented` -> `verified`; detalle y evidencia requerida en [knowledge/validacion.md](../knowledge/validacion.md).
7. **Contexto presupuestado (CCDD Nivel 2)**: Antes de implementar una tarea delegada, ensambla su contexto con `python scripts/assemble_context.py ccdd/context.json "<tarea>"` (añade `-v` para el contexto completo). El reporte de slots/guardrails que imprime forma parte de la evidencia y se pega en `.agents/logs/<task>-REPORT.md`. Un guardrail `on_fail: abort` (exit 2) bloquea la tarea: no continúes hasta resolverlo. La doc del ensamblador vive en el script; aquí solo se referencia por ruta.

Este archivo es la fuente única de verdad para cualquier agente, sea cual sea la herramienta que lo hospeda. `.cursorrules`, `.github/copilot-instructions.md`, `.clinerules` y `.windsurfrules` en la raíz del repo son punteros delgados a este mismo archivo (convención de cada herramienta), no una copia — si hay divergencia, este archivo manda. Para un humano que revisa lo que produjo un agente, ver [knowledge/supervision-humana.md](../knowledge/supervision-humana.md).
